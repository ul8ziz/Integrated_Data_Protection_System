# استكشاف: عدم ظهور إشعارات المدير عند التثبيت على جهاز آخر

## لماذا لا تظهر الإشعارات للمدير؟

عند تثبيت المشروع على جهاز جديد، قد لا تظهر قائمة **Security Alerts** أو تظهر فارغة. الأسباب المحتملة والحلول:

---

### 1. تسجيل الدخول كمدير (Admin)

- **الوضع:** تبويب "التنبيهات" وواجهة الإشعارات تظهر **فقط** للمستخدمين بصلاحية **Admin**.
- **الحل:**
  - سجّل الدخول بحساب مدير.
  - على التثبيت الأول يُنشأ حساب افتراضي:
    - **Username:** `admin`
    - **Password:** `admin123`
  - بعد الدخول، تأكد أن دورك يظهر "Admin" في الواجهة، ثم افتح تبويب **Security Alerts**.

---

### 2. MongoDB غير مشغّل أو غير مثبت

- **الوضع:** الإشعارات والسياسات والمستخدم الافتراضي يُخزَّنون في **MongoDB**. إذا كان MongoDB غير مشغّل على الجهاز الجديد:
  - لا يُنشأ المستخدم الافتراضي `admin`.
  - لا تُنشأ السياسة الافتراضية.
  - لا يُحفظ أي تنبيه عند حدوث انتهاك.
- **الحل:**
  - تثبيت MongoDB من: https://www.mongodb.com/try/download/community
  - تشغيل خدمة MongoDB:
    - **Windows:** Services → MongoDB أو: `net start MongoDB`
    - **Linux/Mac:** `sudo systemctl start mongod` أو `mongod --dbpath /path/to/data`
  - إعادة تشغيل تطبيق الـ Backend ثم الدخول كـ `admin` / `admin123`.

---

### 3. لا توجد أحداث أنشأت تنبيهات

- **الوضع:** التنبيهات تُنشأ فقط عند **انتهاك سياسة**، مثلاً:
  - تحليل نص أو ملف يحتوي على بيانات حساسة (أسماء، هواتف، بريد، بطاقات ائتمان، إلخ).
  - إرسال بريد اختباري (Test Email) يحتوي على بيانات حساسة.
- على جهاز جديد، إذا لم يُجرَ أي تحليل أو إرسال بريد يطابق السياسة، ستكون قائمة الإشعارات **فارغة** وهذا طبيعي.
- **الحل:**
  - التأكد من وجود سياسة مفعّلة (السياسة الافتراضية تُنشأ عند أول تشغيل مع MongoDB متصل).
  - تجربة "تحليل نص" من تبويب Analysis بنص يحتوي مثلاً على رقم هاتف أو بريد إلكتروني.
  - أو إرسال بريد اختباري من تبويب Test Email بنفس نوع المحتوى، ثم تحديث قائمة التنبيهات (زر Refresh).

---

### 4. خطأ في المصادقة (401/403)

- **الوضع:** إذا انتهت الجلسة أو كان التوكن غير صالح، طلب `/api/alerts/` يرجع 401 أو 403 ويتم تسجيل الخروج تلقائياً، فتبدو "الإشعارات لا تظهر".
- **الحل:**
  - إعادة تسجيل الدخول كـ Admin.
  - التأكد من فتح الواجهة من **نفس عنوان التطبيق** (نفس الـ origin) الذي يعمل عليه الـ Backend؛ لا استخدام localhost من جهاز وواجهة من جهاز آخر بدون إعداد CORS/Proxy إن لزم.

---

### 5. إعدادات البيئة على الجهاز الجديد

- **الوضع:** إذا كان الـ Backend يقرأ إعدادات من ملف `.env` أو متغيرات بيئة، قد تختلف الإعدادات على الجهاز الجديد (مثلاً عنوان MongoDB أو اسم قاعدة البيانات).
- **الحل:**
  - نسخ أو إنشاء ملف `.env` في مجلد `backend` مع على الأقل:
    - `MONGODB_URL` (مثلاً `mongodb://localhost:27017`)
    - `MONGODB_DB_NAME` (مثلاً `Secure_db`)
  - التأكد أن الـ Backend يبدأ بدون أخطاء اتصال MongoDB (مراجعة سجلات التشغيل أو صفحة `/health`).

---

## التحقق السريع

1. فتح `/health` في المتصفح (أو من واجهة المراقبة إن وُجدت) والتأكد أن **MongoDB: connected**.
2. تسجيل الدخول بـ **admin** / **admin123** والتأكد أن الدور "Admin" يظهر.
3. فتح تبويب **Security Alerts** والنقر على **Refresh**.
4. إن كانت القائمة فارغة: تنفيذ تحليل نص أو إرسال بريد اختباري يحتوي على بيانات حساسة ثم **Refresh** مرة أخرى.

إذا استمرت المشكلة، راجع سجلات الـ Backend (`logs/app.log`) وأي رسائل خطأ في وحدة تحكم المتصفح (F12) عند فتح تبويب التنبيهات أو النقر على Refresh.

---

## نافذة "Policy Violation" لا تظهر على المتصفح عند التثبيت على جهاز آخر

عند التثبيت على جهاز آخر، نافذة **Policy Violation** (إشعار انتهاك السياسة) قد لا تظهر رغم وجود تنبيهات. الأسباب والحلول:

### 1. الواجهة والـ API على منفذ/مضيف مختلف

- **الوضع:** المتصفح يفتح الواجهة من عنوان مثل `http://192.168.1.10/` (منفذ 80) بينما الـ Backend يعمل على `http://192.168.1.10:8000`. طلبات الإشعارات تُرسل إلى المنفذ 80 فتفشل أو لا تصل للـ API.
- **الحل (أ):** استخدام **الوكيل العكسي (Reverse Proxy)** بحيث يوجّه نفس المضيف/المجال مسار `/api` إلى الـ Backend (مثلاً إلى المنفذ 8000). عندها لا حاجة لتعديل الواجهة.
- **الحل (ب):** تحديد عنوان الـ API يدوياً في الواجهة بإحدى الطريقتين:
  1. **وسم meta في `index.html`** (قبل تحميل `script.js`):
     ```html
     <meta name="api-base" content="http://IP_الجهاز:8000">
     ```
     مثال: `<meta name="api-base" content="http://192.168.1.10:8000">`
  2. **أو في الصفحة نفسها من وحدة التحكم (F12):** قبل تسجيل الدخول تنفيذ:
     ```javascript
     window.API_BASE = 'http://IP_الجهاز:8000';
     ```
     ثم تحديث الصفحة وتسجيل الدخول كـ Admin.

### 2. عدم تسجيل الدخول كـ Admin

- إشعار **Policy Violation** يعمل فقط لحساب **Admin**. المستخدمون العاديون لا يشغّلون است轮询 التنبيهات.
- **الحل:** تسجيل الدخول بحساب مدير (مثلاً `admin` / `admin123`) والتأكد من ظهور الدور "Admin" في الواجهة.

### 3. انتهاء الجلسة أو فشل المصادقة (401)

- إذا أعاد الـ API 401، يتوقف است轮询 التنبيهات ويُعاد توجيه المستخدم لتسجيل الدخول.
- **الحل:** إعادة تسجيل الدخول. إن استمر 401، التحقق من أن الـ API يعمل وأن عنوان `api-base` صحيح (انظر 1).

### 4. CORS أو خطأ شبكة

- إذا كان الـ API على مضيف أو منفذ مختلف ولم يُضبط CORS أو `api-base` بشكل صحيح، قد تفشل الطلبات ويظهر تحذير في وحدة التحكم (F12).
- **الحل:** في الـ Backend التأكد من إعداد CORS ليشمل عنوان الواجهة (مثلاً `origins=["http://192.168.1.10"]`). وفي الواجهة التأكد من ضبط `api-base` كما في الفقرة 1.

### التحقق من عمل إشعار Policy Violation

1. فتح وحدة التحكم (F12) في المتصفح.
2. تسجيل الدخول كـ **Admin**.
3. البحث عن السطر: `[Policy Violation] Admin notification polling started; API base: ...` — إذا ظهر فاست轮询 يعمل.
4. إذا ظهر تحذير مثل `[Policy Violation] Failed to fetch recent alerts:` فمراجعة الرابط (URL) والـ CORS وضبط `api-base` كما أعلاه.
